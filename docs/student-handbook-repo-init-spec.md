# 內湖國小學生手冊 repo 初始化規格

## 一、文件目的

本文件是為未來建立 `nhps-student-handbook` 獨立 repo 所準備的初始化規格。目標是先定義目錄結構、資料格式、來源追蹤、MVP 範圍、風險控管與驗收標準，作為後續開案施工圖。

本文件不代表立即實作。現階段不建立新 repo、不撰寫前台程式、不建立 JSON 資料檔、不導入 Google Sheets / GAS / Cloudflare Worker / LINE Bot，也不建立任何 secret 或 token。

## 二、repo 基本資訊規劃

| 項目 | 規劃 |
| --- | --- |
| repo 名稱 | `nhps-student-handbook` |
| 建議 GitHub Pages URL | `https://smallcannon-arch.github.io/nhps-student-handbook/` |
| 專案中文名稱 | 內湖國小學生手冊暨家長校園生活入口 |
| 主要受眾 | 家長為主，高年級學生為輔 |
| 第一版定位 | 靜態前台 MVP |
| 第一版資料來源 | repo 內 `data/*.json` 公開資料 |
| 第一版維護方式 | 先以手動編輯 JSON 與文件審核為主 |

第一版目標是驗證家長入口、學生友善文字、六大分類、搜尋與手機閱讀體驗，不是一次完成完整行政後台或 LINE 查詢系統。

## 三、為何獨立 repo

1. 避免與教師手冊部署混線：教師手冊已經有既有 GitHub Pages、Worker、LINE Bot 與後台管理流程；學生手冊應有獨立部署節奏。
2. 避免資料權限混線：學生手冊面向家長與學生，資料來源、審核責任與公開邊界不同於教師手冊。
3. 避免 LINE Bot / Worker / 後台設定混線：第一版不導入這些元件，未來若導入也應使用獨立設定與明確邊界。
4. 讓學生手冊可以獨立發展家長入口與學生友善設計：前台語氣、分類與卡片欄位應以家長生活情境為中心。
5. 教師手冊 repo 僅保留企劃歷程與規格文件：正式產品檔案、靜態前台與資料檔應移至新 repo。

## 四、校網公開資料盤點納入原則

校網首頁為 `https://www.nhps.hc.edu.tw/nss/p/index`。公開盤點顯示，校網已有若干可作為學生手冊資料來源的入口，但多數生活流程仍需處室確認後才能轉成家長友善卡片。

校網公開基本資訊：

| 項目 | 內容 |
| --- | --- |
| 校名 | 新竹市內湖國小／新竹內湖國小 |
| 電話 | 03-5373184 |
| 傳真 | 03-5371576 |
| 地址 | 300 新竹市香山區內湖路109號 |
| 主要選單 | 探索內湖、處室簡介、處室職掌、重大議題專區、課務專區、數位精進、入學專區、學生家長會、幼兒園地、校內專區、線上報修 |

### 1. 可作為第一版資料來源的校網入口

- 校網首頁。
- 行事曆。
- 評量時間表。
- 班親會或校務活動相關公告。
- 開學／休業式相關資訊。
- 處室聯絡、處室簡介、處室職掌。
- 反霸凌專區。
- 性別平等教育專區。
- 健康中心、午餐、圖書館／圖資組等單位入口。
- 入學專區。
- 活動公告。
- 交通安全專區。

這些資料可作為卡片來源連結或背景依據，但前台卡片不直接複製公告全文，應採「家長友善摘要 + 原始來源連結」。

### 2. 校網有入口但需處室確認的資料

- 行事曆中的日常作息、特殊日放學與班親會資訊。
- 評量時間表與成績評量說明。
- 處室聯絡與服務項目。
- 健康中心一般服務與學生受傷處理。
- 午餐供應、訂退餐與特殊飲食。
- 圖書館借還書與圖資服務。
- 交通安全與上放學動線。
- 反霸凌與性別平等教育相關內容。

這些資料可以列入第一版盤點，但不得在未查核時標示為正式定稿。

### 3. 校網目前未找到、需人工提供的資料

- 請假與出缺席完整流程。
- 日常上學與放學規則。
- 路隊、家長接送、安親班接送、雨天放學。
- 學生受傷、發燒、服藥、健康中心通知流程。
- 午餐退餐、特殊飲食、餐具、飲水。
- 行動載具與智慧手錶規範。
- 失物招領流程。
- 學生申訴正式流程。
- 家長常用表單總入口。

這些卡片應標示 `needs_office_review: true`，來源狀態應為「校網未找到，需人工提供」或「校網有入口但需處室確認」。

### 4. 不建議直接納入學生手冊的資料

- 舊年度 PDF 或舊公告。
- 未確認有效性的歷史資料。
- 霸凌、性平文件中的法規文字直接改寫成認定教學。
- 個人聯絡資料、個別公告內的人員資料。
- 班級課表整批抓入手冊。
- 幼兒園、公托家園資料混入國小部學生手冊。

霸凌、性平資料只能整理成求助與依法處理方向，不做認定教學，不讓家長或學生自行判斷事件成立與否。

來源 URL、來源狀態、查核日期、查核責任與處室確認狀態應納入資料結構，避免內容失去來源或過期仍被當成正式資訊。

## 五、第一版技術原則

1. 使用 GitHub Pages。
2. 第一版只用靜態檔案。
3. 第一版內容資料放在 repo 內 `data/*.json`。
4. 第一版不接 Google Sheets。
5. 第一版不接 GAS。
6. 第一版不接 Cloudflare Worker。
7. 第一版不做 LINE Bot。
8. 第一版不建立 secret / token。
9. 第一版不處理任何個資。
10. 第一版先驗證內容架構、搜尋、手機閱讀與家長入口是否好用。

GitHub Pages 適合發布 HTML、CSS、JavaScript 與靜態資料檔。第一版不需要伺服器端程式，也不需要 Worker deploy。

## 七、第一版前台頁面規格

本章只規劃頁面功能，不撰寫程式。

前台應包含：

1. 頁首：校名、系統名稱、副標。
2. 搜尋框。
3. 六大入口：
   - 今日作息
   - 請假與出缺席
   - 上放學與接送
   - 健康與午餐
   - 學習與活動
   - 找人幫忙
4. 常用卡片。
5. 分類 chip。
6. 卡片式結果。
7. 最近更新。
8. 處室聯絡資訊。
9. 頁尾聲明。
10. 原始來源連結或來源說明。

卡片需顯示：

- 標題。
- 給家長。
- 學生這樣做。
- 處理步驟。
- 找誰。
- 相關連結。
- 原始來源。
- 來源狀態。
- 負責處室。
- 更新日期。

敏感議題卡片應以求助、保護與權責窗口為主，不呈現事件認定教學。

## 八、JSON 資料格式規格

`data/handbook-items.json` 建議使用陣列，每筆代表一張公開卡片。

| 欄位 | 用途 | 第一版必填 |
| --- | --- | --- |
| `id` | 卡片唯一識別碼。 | 是 |
| `title` | 卡片標題。 | 是 |
| `category` | 六大分類之一。 | 是 |
| `audience` | 主要受眾，例如家長、學生、家長與學生。 | 是 |
| `grade_level` | 適用年級或年段。 | 是 |
| `situation` | 使用者生活情境。 | 是 |
| `summary_parent` | 給家長看的摘要。 | 是 |
| `summary_student` | 給學生看的短句與求助導向文字。 | 視卡片而定；學生可自行理解或行動的卡片必填 |
| `steps` | 建議處理步驟。 | 是 |
| `contact_unit` | 負責或可詢問的處室／角色。 | 是 |
| `links` | 相關公開連結。 | 否 |
| `forms` | 相關公開表單。 | 否 |
| `legal_basis` | 法規或校內規定來源名稱。 | 敏感或規範類卡片必填 |

| `status` | 草稿、待處室確認、已發布、待更新等狀態。 | 是 |
| `pinned` | 是否常用置頂。 | 是 |
| `updated_at` | 卡片更新日期。 | 是 |

## 九、來源資料規格

`data/sources.json` 用於保存來源清單、來源可用性、查核日期與是否納入 MVP。

| 欄位 | 用途 |
| --- | --- |
| `id` | 來源唯一識別碼。 |
| `title` | 來源標題。 |
| `url` | 公開來源 URL；人工提供資料可留空或填內部來源說明，但不得放個資。 |
| `source_type` | 來源類型。 |
| `related_categories` | 對應分類。 |
| `related_item_ids` | 對應卡片 ID。 |
| `source_status` | 來源狀態。 |
| `checked_at` | 最近查核日期。 |
| `checked_by_role` | 查核角色，例如教務處、學務處、輔導室。 |
| `needs_office_review` | 是否仍需處室確認。 |
| `review_note` | 查核備註，不放個資或個案。 |
| `include_in_mvp` | 是否列入第一版 MVP。 |

`source_status` 建議值：

- 校網已有來源。
- 校網有入口但需處室確認。
- 校網未找到，需人工提供。
- 舊資料，需查核後才能使用。
- 不建議納入。

`source_type` 建議值：

- 校網首頁。
- 行事曆。
- PDF。
- 公告。
- 處室頁。
- 重大議題專區。
- 表單。
- 人工提供。
- 其他。

## 十、分類資料規格

`data/categories.json` 用於前台六大入口與分類 chip。

六大分類：

1. 今日作息
2. 請假與出缺席
3. 上放學與接送
4. 健康與午餐
5. 學習與活動
6. 找人幫忙

| 欄位 | 用途 |
| --- | --- |
| `id` | 分類 ID，例如 `daily`、`absence`。 |
| `label` | 前台顯示名稱。 |
| `description` | 分類說明，給家長理解入口用途。 |
| `icon` | 圖示名稱或圖示檔路徑。 |
| `sort_order` | 前台排序。 |

## 十一、聯絡資訊資料規格

`data/contacts.json` 應使用一般處室與角色，不放私人 email、個人手機或個資。

建議包含：

- 教務處
- 學務處
- 總務處
- 輔導室
- 健康中心
- 午餐承辦
- 導師

| 欄位 | 用途 |
| --- | --- |
| `unit` | 單位或角色名稱。 |
| `description` | 單位服務範圍。 |
| `phone_note` | 聯絡方式說明，例如「請先洽學校總機轉接」。 |
| `service_note` | 服務提醒，例如「課務、評量、學籍相關問題」。 |
| `sort_order` | 前台排序。 |

## 十三、內容撰寫規範

未來 `docs/content-guidelines.md` 應包含下列規範：

1. 家長版文字：清楚、具體、能照著做。優先回答「我現在該做什麼、找誰、看哪裡」。
2. 學生版文字：短句、行動句、求助導向。避免使用學生不易理解的行政詞。
3. 敏感議題：不做認定教學、不提供法律結論、不寫個案。
4. 法規內容：未查核不得上線；涉及霸凌、性平、申訴、獎懲與個資時需權責單位查核。
5. 校網資料：不直接複製公告全文，應改寫成家長友善摘要並保留來源連結。
6. 每張卡片需有 `owner`、`reviewer`、`updated_at`、`source_status`。
7. 不放學生姓名、家長姓名、班級座號、電話、健康資訊。
8. 不放 secret / token。
9. 連結需可驗證且定期檢查。

## 十四、MVP 驗收標準

未來 `docs/mvp-acceptance.md` 應包含下列標準：

1. 手機 390px 寬度無橫向溢出。
2. 搜尋「請假」「放學」「接送」「健康中心」「午餐」「霸凌」「性平」可命中相關卡片。
3. 家長能在 10 秒內找到 6 大入口。
4. 每張卡片都有 `owner`、`reviewer`、`updated_at`、`source_status`。
5. 校網已有來源的卡片應有 `source_urls` 或 `sources.json` 對應。
6. 需處室確認的卡片不得標示為正式定稿。
7. 敏感議題不做個案判斷。
8. 無學生個資、無健康個資、無 token / secret。
9. 第一版不依賴外部 API。
10. GitHub Pages 可正常開啟。
11. 只包含靜態檔案，不需要 Worker deploy。

## 十五、第一版不做事項

- 不做後台。
- 不做 Google Sheets。
- 不做 GAS。
- 不做 Cloudflare Worker。
- 不做 KV。
- 不做 LINE Bot。
- 不做 LINE Rich Menu。
- 不做 Push API。
- 不做主動推播。
- 不做通報系統。
- 不蒐集個資。
- 不處理霸凌／性平個案資料。
- 不提供法律判斷。

## 十六、後續導入路線

| 階段 | 內容 | 啟動條件 |
| --- | --- | --- |
| Phase 0 | repo 初始化與文件 | repo 名稱、公開範圍、資料欄位與來源追蹤規格確認。 |
| Phase 1 | 靜態前台與 JSON 資料 | 第一批分類、資料欄位與首頁互動規格確認。 |
| Phase 2 | 內容盤點與 40 張卡片 | 40 張卡片的 owner、reviewer、source_status 與是否需處室確認完成初盤。 |
| Phase 3 | Google Sheets 資料化 | JSON 欄位穩定，且有非技術維護者需要共同維護內容。 |
| Phase 4 | 後台管理 | Google Sheets 欄位穩定，且人工改 JSON 已不符合維護需求。 |
| Phase 5 | Cloudflare Worker / cache | 前台讀取效能、跨來源或後台 proxy 需求明確。 |
| Phase 6 | LINE Rich Menu / LINE Bot | 前台內容穩定、敏感議題文字查核完成，且明確只做查詢既有內容。 |

每一階段需獨立稽核後再進入下一階段，不直接全部開做。

## 十七、風險控管

| 風險 | 控管方式 |
| --- | --- |
| 與教師手冊混線 | 正式系統使用獨立 repo；教師手冊 repo 只保留企劃與規格文件。 |
| MVP 範圍過大 | 第一版只做靜態前台、JSON 與 40 張優先卡片，不導入後台、Worker 或 LINE。 |
| 敏感議題文字越界 | 霸凌、性平、申訴、健康等卡片需 reviewer 查核；只寫求助與保護，不做判斷。 |
| 個資外洩 | 不放學生姓名、家長姓名、班級座號、電話、健康資訊或個案內容。 |
| 法規過期 | `legal_basis`、`source_checked_at`、`reviewer` 必填；定期查核。 |
| 連結失效 | `sources.json` 集中管理來源，建立定期連結檢查清單。 |
| 手機閱讀不佳 | MVP 驗收納入 390px 寬度與常用流程 10 秒查找。 |
| JSON 結構日後轉 Sheets 困難 | 欄位設計對齊未來 Google Sheets 主表，避免自由格式過多。 |
| 校網資料來源過期或失連 | 每筆來源記錄 `source_status` 與 `checked_at`，舊資料不得直接上線。 |
| 校網公告直接複製造成語氣不適合家長 | 改寫為家長友善摘要，保留原始來源連結，不複製公告全文。 |
