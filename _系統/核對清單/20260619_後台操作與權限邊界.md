- 哪些操作屬於高風險治理行為。
- 前台與後台入口的安全邊界。
- 哪些項目已完成驗收，哪些項目需另案處理。

## 3. 角色定義

目前後台角色以 Google 帳號登入後，由系統依資料層使用者角色判斷。

- editor：一般後台編輯者，可維護教師手冊資料卡與前台顯示狀態。
- admin：系統管理者，可執行高風險治理操作。
- reviewer：legacy 相容角色，暫等同 editor，不作為主要流程角色。

若未來要新增完整權限管理頁、調整角色定義或新增角色，應另開 PR，並重新進行 UI 與後端權限 QA。

## 4. editor 權限邊界

editor 可執行：

- 登入後台。
- 編輯資料卡內容。
- 儲存資料卡。
- 複製資料卡。
- 修改 visible。
- 上架 / 下架資料卡。

editor 不可執行：

- 移置回收桶。
- 從回收桶還原。
- 刪除資料。
- 管理使用者權限。
- 執行其他高風險治理操作。

editor 可維護一般流程性、入口性、提醒性資料，但不得在後台填入學生個資、個案內容、成績、校安細節、採購工程細節或未經確認的高風險資訊。

## 5. admin 權限邊界

admin 具 editor 權限，並可執行高風險治理操作。

admin 可執行：

- editor 可執行的所有操作。
- 移置回收桶。
- 從回收桶還原。
- 刪除或執行高風險治理操作。
- 未來可管理使用者權限。

admin 操作原則：

- 高風險操作應限於可確認、可回復或可稽核的資料。
- 不應對正式重要資料進行未規劃的破壞性測試。
- 若需測試刪除、還原或其他治理操作，應使用明確測試前綴資料。
- 若高風險操作涉及跨角色權限調整，需另開 PR 並重新 QA。

## 6. reviewer legacy 相容定位

reviewer 僅作 legacy 相容角色。

- reviewer 暫等同 editor 等級。
- reviewer 不作為主要流程角色。
- reviewer 不應被視為獨立審核流程的正式角色。
- 未來若要恢復 reviewer 審核流程，需另開 PR 設計完整流程與 QA 矩陣。

## 7. 前台與後台入口邊界

目前入口邊界如下：

- 前台 index.html 不放「後台管理」入口。
- 後台入口由 login.html / admin.html 承擔。
- login.html 可回前台教師手冊。
- admin.html 可回前台。
- prototype.html 不恢復、不作為側門入口。

此邊界的目的：

- 避免前台使用者直接看到後台入口。
- 保留已授權維護者從後台回到前台檢查成果的便利性。
- 避免將後台恢復與前台導覽策略混在同一輪功能變更中。

若未來要在前台新增後台入口，應另開 PR，並明確評估產品需求、權限風險與入口文案。

## 8. 資料操作風險分級

低風險操作：

- 編輯一般流程文字。
- 更新一般連結標籤。
- 補充全體教師可查閱的提醒。
- 調整非敏感分類或排序。

中風險操作：

- 修改 visible。
- 上架 / 下架資料卡。
- 複製資料卡後再整理內容。
- 批次調整前台可見資料。

高風險操作：

- 移置回收桶。
- 從回收桶還原。
- 刪除或停用資料。
- 使用者權限管理。
- 涉及學生個資、個案、成績、校安細節、採購工程細節或未經確認的高風險資料。

後台不得放入：

- 學生個資。
- 個案內容。
- 成績。
- 校安細節。
- 採購工程細節。
- 未經承辦處室確認的高風險資訊。
