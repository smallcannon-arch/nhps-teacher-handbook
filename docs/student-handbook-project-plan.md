# 內湖國小學生手冊暨家長校園生活入口系統企劃書

## 一、企劃名稱

正式名稱：內湖國小學生手冊暨家長校園生活入口系統。

簡短副標：讓家長找得到、學生看得懂、學校維護得起的校園生活指引。

本企劃以既有「內湖國小教師手冊」之網站、資料維護與 LINE 查詢經驗為基礎，規劃一套面向家長與高年級學生的校園生活資訊入口。系統初期以企劃、資料盤點與內容架構為主，後續再依實際需求分階段建置前台、後台、API 與 LINE 查詢服務。

## 二、企劃緣起

學校日常運作中，許多重要資訊分散在校網公告、導師通知、處室表單、紙本資料與口頭說明之中。對熟悉學校行政流程的教職員而言，這些資訊各有脈絡；但對家長與學生而言，常常需要先知道「該找誰」或「該看哪裡」，才能開始解決問題。

家長常見需求包含課表、上放學時間、接送方式、請假流程、健康中心、午餐、活動資訊、社團、常用表單，以及孩子在校遇到狀況時可以向誰求助。高年級學生也逐漸需要能自行理解的校園生活指引，知道遇到身體不舒服、東西遺失、人際衝突、被同學欺負或其他不舒服情境時，可以怎麼做、可以找誰幫忙。

因此，本案希望建立一套可長期維護、可搜尋、可更新的學生手冊與家長入口系統，而不只是一次性 PDF。手冊內容應兼具制度說明與生活引導，讓家長快速掌握學校流程，也讓學生知道學校裡有可以協助自己的大人與單位。

## 三、目標受眾

1. 家長：主要使用者。家長需要快速查詢孩子每日校園生活、請假、接送、健康、午餐、活動、表單與求助管道等資訊。入口設計應以家長常見情境為起點，而非以行政組織分工為起點。
2. 高年級學生：輔助使用者。高年級學生可以透過簡短、生活化的文字理解基本規範與求助方式。內容不要求學生自行判斷複雜事件是否成立，而是引導學生找可信任的大人、導師、學務處、輔導室或家長。
3. 導師與行政人員：輔助查詢者與內容維護者。導師與行政人員可用同一套入口快速回覆家長常見問題，也可透過後台或資料表維護內容，降低資訊落差與重複回覆成本。

## 四、企劃目標

1. 降低家長找資料成本，讓常見問題能在手機上快速查到。
2. 統一校園生活流程說明，避免不同來源資訊不一致。
3. 讓高年級學生知道遇到問題可以找誰、可以怎麼求助。
4. 減少導師與行政人員重複回覆相同問題的負擔。
5. 讓手冊內容可維護、可查核、可更新，而不是只在學年初發布後逐漸過期。
6. 兼顧法規正確性、學生權益與個資安全，敏感議題只呈現求助管道與學校依法處理原則，不做個案判斷。

## 五、需求分析

家長使用手冊時，通常不是從法規或處室名稱開始，而是從生活情境開始。本系統應優先支援以下常見情境：

- 今天孩子生病不能到校，需要知道請假方式、是否需要證明、是否影響午餐或課務。
- 孩子遲到或需要早退，需要知道校門進出、家長接回、導師與學校聯繫方式。
- 不知道今天幾點放學，或遇到特殊活動、考試、天候因素時需要確認放學安排。
- 不知道接送區、路隊位置或安親班接送規範。
- 孩子在校受傷或身體不舒服，需要知道健康中心處理原則、通知流程與後續追蹤。
- 午餐、退餐、特殊飲食或餐具飲水等生活問題需要清楚說明。
- 不確定手機、智慧手錶或其他行動載具能不能帶到學校，以及在校使用規範。
- 孩子東西不見，需要知道失物招領、班級詢問與學務處協助方式。
- 孩子說被同學欺負，需要知道可以先向誰反映、學校會如何啟動協助，而不是要求家長自行判斷是否構成特定事件。
- 疑似霸凌、疑似性平事件不知道找誰，需要有清楚、保護當事人、避免揭露個資的求助引導。
- 想查社團、圖書館、活動、表單、班親會、評量或其他校園生活資訊。

## 六、手冊內容架構

### 1. 快速開始：家長一定要知道的 20 件事

目的：讓新生家長、轉學生家長或臨時需要查詢的家長快速掌握最常用資訊。

建議內容：上學時間、放學時間、請假方式、遲到早退、接送方式、健康中心、午餐、常用聯絡單位、失物招領、行動載具、校園安全、找人幫忙、常用表單、校網與公告來源等。

家長問法例句：我剛接觸學校，最先要知道哪幾件事？

### 2. 每日作息與課表

目的：整理學生每日在校作息，降低家長對時間安排的疑問。

建議內容：課表查詢、晨間時間、午餐午休、放學時段、半日課、課後活動、特殊日課程調整、考試或活動日提醒。

家長問法例句：今天孩子幾點上學、幾點放學？

### 3. 請假與出缺席

目的：清楚說明家長請假、學生遲到早退、出缺席通知與後續補課務的基本流程。

建議內容：病假、事假、公假、臨時請假、遲到、早退、連續請假、出缺席紀錄、導師通知、可能需要的文件或表單。

家長問法例句：我今天孩子不能到校，要怎麼請假？

### 4. 上放學、路隊與校園安全

目的：協助家長理解接送、路隊、校門進出與校園安全規範。

建議內容：上學動線、放學動線、路隊安排、安親班接送、家長接送區、雨天放學、訪客入校、交通安全、緊急狀況通知。

家長問法例句：雨天我要在哪裡接孩子？

### 5. 健康中心、緊急傷病與傳染病

目的：說明學生在校身體不舒服、受傷、服藥或疑似傳染病時的處理原則。

建議內容：健康中心使用、受傷處理、發燒或身體不適、家長通知、就醫建議、在校服藥、傳染病通報與請假、復課注意事項。此章不放健康個資與個案內容。

家長問法例句：孩子在學校受傷，學校會怎麼通知？

### 6. 午餐、飲水與校園生活

目的：整理學生每日生活所需的午餐、飲水、環境衛生與校園服務資訊。

建議內容：午餐供應、退餐或異動、特殊飲食協助、餐具、飲水、環境整潔、失物招領、校園空間使用。

家長問法例句：孩子午餐要退餐或有特殊飲食需求，要怎麼處理？

### 7. 學習、評量與活動

目的：讓家長了解學生學習活動、評量安排與校園參與機會。

建議內容：定期評量、平時學習、作業與補交、社團、校隊、校外教學、圖書館、競賽、班親會、學校活動。

家長問法例句：我想查社團、評量或校外教學資訊，要看哪裡？

### 8. 數位生活、行動載具與網路安全

目的：說明學生使用行動載具、網路資源與數位設備時的基本規範。

建議內容：手機與智慧手錶、平板或學習載具、校內使用規範、網路安全、數位禮節、家長協助事項。

家長問法例句：孩子可以帶手機或智慧手錶到學校嗎？

### 9. 人際互動、霸凌防制、性別平等與求助

目的：提供學生與家長遇到人際衝突、疑似霸凌、疑似性平事件或不舒服情境時的求助指引。

建議內容：找導師、找學務處、找輔導室、告訴家長、保護自己、記錄需要協助的事實、學校依法處理原則。此章不設計成事件認定教學，不要求學生或家長自行判斷是否成立霸凌或性平事件。

家長問法例句：孩子說被同學欺負或覺得不舒服，我可以找誰？

### 10. 學生權利、獎勵管教、申訴與重要規定

目的：以家長與學生可理解的方式整理重要權利義務與校內規定。

建議內容：學生獎勵、輔導管教、學生申訴、輔導協助、校規摘要、個資保護、校園安全規定與相關法規連結。法規內容需後續由權責單位查核。

家長問法例句：如果我對學生權益、獎懲或申訴流程有疑問，可以從哪裡查起？

## 七、家長入口設計

首頁建議採六大入口，讓家長從常見情境進入，而不是先理解處室分工。

1. 今日作息：查詢上學時間、放學時間、課表、半日課、活動日、考試日與特殊作息提醒。
2. 請假與出缺席：查詢病假、事假、公假、遲到、早退、臨時請假、連續請假與出缺席紀錄相關說明。
3. 上放學與接送：查詢路隊、接送區、安親班接送、雨天放學、家長入校、交通安全與校園門禁。
4. 健康與午餐：查詢健康中心、學生受傷、發燒、傳染病、在校服藥、午餐、退餐、特殊飲食、飲水與餐具。
5. 學習與活動：查詢評量、作業、社團、圖書館、校外教學、競賽、班親會、校隊與學校活動。
6. 找人幫忙：查詢導師、教務處、學務處、總務處、輔導室、健康中心、午餐承辦與其他求助管道；敏感事件以安全、保護與轉介為核心。

## 八、高年級學生友善設計

學生友善內容應使用短句，並盡量以「我可以怎麼做」的格式呈現。文字應少用行政詞與法規詞，避免讓學生覺得只有懂制度的人才看得懂。

對於疑似霸凌、疑似性平事件或其他敏感情境，不要求學生自行判斷事件是否構成特定法律或校園事件類型。手冊只引導學生找可信任的大人、導師、學務處、輔導室或家長，並提醒學生保護自己、不要獨自承擔。

每張重要卡片可增加「學生這樣做」小段落，讓高年級學生在手機上能快速找到下一步。

示例文字：

1. 我身體不舒服。  
   如果你在學校覺得頭痛、肚子痛、受傷或很不舒服，可以先告訴老師，請老師協助你到健康中心。不要自己忍耐，也不要自己離開校園。健康中心或老師會視情況聯絡家長。

2. 我東西不見了。  
   先回想最後一次看到物品的地方，告訴導師，也可以到班上、使用過的教室或失物招領處詢問。找到別人的物品時，請交給老師或學校協助處理。

3. 我覺得被同學欺負或不舒服。  
   如果有人讓你害怕、難過、不舒服，或你不知道該怎麼辦，請先找你信任的大人，例如導師、輔導室、學務處或家長。你不需要自己判斷這件事叫什麼，先讓大人知道，學校會協助保護你並了解情況。

## 九、法規與政策依據章節

本章節只規劃需查核的法規與政策主題，不在企劃階段自行編寫條文內容。後續正式上線前，應由權責處室或指定查核者確認最新版本與校內適用方式。

法規查核原則：法規摘要不得未查核即上線。涉及霸凌、性平、學生申訴、獎懲、輔導管教、個資等敏感內容，應由權責處室或具相關專業人員協助查核。前台文字應以求助流程、權責窗口與學生權益保護為主，不在手冊中做個案認定，也不提供法律結論。法規連結與查核日期應納入資料欄位或後台管理，讓維護者能追蹤內容是否仍為最新可用版本。

| 主題 | 手冊中要呈現的方向 | 需要後續查核的文件或法規 | 前台呈現時的注意事項 |
| --- | --- | --- | --- |
| 校園霸凌防制 | 呈現求助管道、學校受理與協助原則、保護當事人與避免二度傷害。 | 教育部及地方政府校園霸凌防制相關規定、校內處理流程。 | 不寫成霸凌認定教學，不要求家長或學生自行判斷成立與否。 |
| 性別平等教育 | 呈現尊重身體界線、遇到不舒服事件的求助方式、學校依法處理原則。 | 性別平等教育相關法規、地方政府與校內性平處理流程。 | 不放個案，不揭露身分，不做法律判斷，只引導求助。 |
| 學生獎懲與輔導管教 | 讓家長理解獎勵、提醒、輔導、管教與申復或溝通管道。 | 校內學生獎懲、輔導與管教相關規定。 | 避免只列處罰，應呈現教育目的、溝通與輔導資源。 |
| 學生申訴 | 說明學生或家長對權益事項有疑問時可循的反映與申訴管道。 | 學生申訴相關規定、校內申訴流程。 | 文字要清楚、不中斷求助，不把申訴描述成對立程序。 |
| 學生輔導 | 說明輔導室可提供的支持、轉介與家校合作方式。 | 學生輔導相關法規、校內輔導資源與轉介流程。 | 不蒐集或呈現輔導個案內容，不公開健康或家庭隱私。 |
| 緊急傷病與健康中心 | 說明在校受傷、生病、發燒、服藥與家長通知原則。 | 學校緊急傷病處理、健康中心作業與相關衛生規範。 | 不放健康個資；只呈現一般流程與聯絡方式。 |
| 午餐與學校衛生 | 說明午餐供應、退餐、特殊飲食協助與校園衛生原則。 | 學校午餐、食品安全、校園衛生相關規定。 | 避免放供應商或個別學生飲食資料；異動流程要明確。 |
| 個人資料保護 | 說明公開手冊不放個資，後台與資料表需控管權限。 | 個人資料保護相關法規、校內資料使用規範。 | 前台不暴露寫入端點，不顯示學生、家長或個案可識別資訊。 |

特別提醒：霸凌與性平內容不得寫成認定教學，不讓家長或學生自行判斷成立與否。手冊應呈現求助管道、學校依法處理、保護當事人、不揭露個資與避免二度傷害等原則。

## 十二、技術架構規劃

本案可依教師手冊既有經驗分階段規劃，但第一版不需一次完成全部技術元件。

非技術版說明：本系統的技術設計原則，是讓家長與學生看到簡單清楚的前台頁面，讓行政端能用表格維護內容，並透過中介 API 提高讀取速度與降低前台直接暴露後台資料的風險。換言之，使用者看到的是一本好查的線上手冊；維護者操作的是一份可整理、可更新、可查核的資料表；技術層則負責把資料穩定、安全地送到前台。

1. GitHub Pages 前台：作為公開手冊入口，提供靜態 HTML、CSS、JavaScript 與可搜尋卡片介面。GitHub Pages 適合公開靜態內容，不承擔後台寫入或機密資料處理。
2. Google Sheets 作為資料源：以工作表維護卡片內容、分類、聯絡單位、表單與法規查核狀態，方便行政人員檢視與協作。
3. GAS 後台寫入：後續若需要後台管理，可由 GAS 接收後台儲存請求並寫入 Google Sheets。
4. Cloudflare Worker API 與快取：可作為前台讀取資料、快取公開資料、代理 GAS 讀取或後台寫入的中介層。此層應清楚區分公開 GET 與需授權 POST。
5. LINE Bot 與 Rich Menu 作為第二階段：等前台內容穩定後，提供家長從 LINE 查詢既有手冊內容。
6. KV 搜尋索引列為後續優化：當 LINE 查詢或前台搜尋需要更快回應時，再評估建立 KV 搜尋索引，不列第一版必做。

簡易架構圖：

```text
家長 / 高年級學生
        |
        v
GitHub Pages 前台
        |
        v
Cloudflare Worker API（第二階段）
        |
        +--> 快取公開資料
        |
        v
Google Sheets 資料源
        ^
        |
GAS 後台寫入（後續階段）

LINE Rich Menu / LINE Bot（第二階段）
        |
        v
只查詢既有手冊內容，不處理個案，不生成判斷
```

## 十三、前台功能規劃

- 首頁：以六大入口與常用卡片呈現家長最常查詢的內容。
- 搜尋：支援生活語彙與同義詞，例如「生病」「病假」「健康中心」可命中相關卡片。
- 分類 chip：依請假、健康、接送、午餐、學習、求助等分類篩選。
- 常用入口：將上放學、請假、健康中心、午餐、找人幫忙等高頻內容置頂。
- 卡片式結果：每張卡片包含情境、家長摘要、學生摘要、處理步驟、負責處室、更新日期與相關連結。
- 最近更新：讓家長知道哪些制度或連結近期有調整。
- 各處室聯絡資訊：以單位與服務項目呈現，避免公開不必要的個人資料。
- 法規與表單下載：提供公開可用的表單與法規依據連結，並標示查核日期。
- 手機版優先：多數家長會用手機查詢，需優先檢查閱讀性、按鈕大小、卡片間距與橫向溢出。
- 無障礙與字級原則：搜尋框與主要文字需有足夠字級與對比；互動元素需可點、可讀、可理解。

## 十四、後台管理規劃

後台管理可沿用教師手冊經驗，等前台資料格式穩定後再實作。

- 新增／編輯卡片：支援新增生活情境、家長摘要、學生摘要、步驟、連結與負責單位。
- 草稿／已發布／待更新：讓尚未查核或過期內容不直接出現在前台。
- 常用置頂：支援首頁或常用入口顯示。
- 欄位防呆：必要欄位不得空白；狀態與有效連結需一致。
- 連結檢查：提醒連結待補、失效或不適合公開的連結。
- 登入逾時提醒：登入狀態失效時應清楚提示，不讓管理者誤以為已儲存。
- 儲存失敗不清空表單：避免管理者白做工。
- 修改紀錄：保留內容變更時間、變更單位與查核狀態。
- 不新增不必要 Sheets 欄位：優先使用既有欄位解決顯示與管理需求，避免小功能造成資料結構膨脹。

## 十五、LINE Bot 與 Rich Menu 第二階段

LINE Bot 與 Rich Menu 不列入第一版實作，應在前台內容穩定、查詢關鍵字與卡片資料成熟後再規劃。

Rich Menu 六格建議：

1. 學生手冊
2. 請假／出缺席
3. 上放學／接送
4. 課表／行事曆
5. 健康／午餐
6. 找人幫忙

LINE Bot 原則：

- 只使用 Reply API。
- 不做 Push API。
- 不主動推播。
- 不全校群發。
- 不處理個案。
- 不儲存 LINE userId。
- 不生成霸凌、性平或法律判斷。
- 敏感詞只回求助與安全引導。
- 回覆內容只來自學生手冊既有公開內容，避免產生未經查核的新建議。

## 十六、資安與個資原則

- 公開手冊只放一般制度與流程。
- 不放個案資料。
- 不放學生個資。
- 不放健康個資。
- 不在 LINE Bot 蒐集事件細節。
- token / secret 只放部署平台 secrets。
- token / secret 不寫入 repo、文件或 log。
- 後台需權限控管，僅限授權維護者使用。
- 前台需避免暴露寫入端點。
- 敏感議題以求助與保護為核心，不要求使用者在公開介面輸入事件細節。

## 十八、MVP 分階段期程

本企劃書目前目標為討論版，不代表立即進入完整系統開發。建議先用企劃書與內容卡片草稿確認方向，再視校內共識進入前台原型與系統化建置。

第 0 階段：企劃與資料盤點。  
完成企劃書、章節架構、資料欄位、第一批卡片主題、法規查核清單與維護分工草案。

第 1 階段：內容卡片草稿。  
依章節撰寫第一批 40 至 60 張卡片，先以 Markdown 或試算表草稿整理，不急著系統化。

第 2 階段：前台靜態原型。  
以靜態前台驗證家長是否能快速找到資訊，優先檢查手機閱讀、搜尋、分類與常用入口。

第 3 階段：Google Sheets 資料化。  
將卡片內容轉為結構化資料，建立 content、categories、contacts、forms、laws 等工作表。

第 4 階段：後台管理。  
建立管理介面，支援新增、編輯、狀態、置頂、連結與更新紀錄。

第 5 階段：Worker API 與快取。  
在資料量、效能或跨來源需求明確後，再加入 Worker API、快取與必要的後台代理。

第 6 階段：LINE Rich Menu / LINE Bot。  
建立 LINE 入口與查詢服務，只回覆既有手冊內容，不主動推播、不處理個案。

討論版以第 0 階段與第 1 階段為主，重點是確認內容範圍、語氣、分類與家長使用路徑。原型版應在內容方向確認後，再進入第 2 階段前台靜態原型。系統版則從第 3 階段以後，才逐步進入 Google Sheets 資料化、後台管理、Worker API 與快取，以及 LINE Rich Menu / LINE Bot。

## 二十、風險控管

| 風險 | 控管方式 |
| --- | --- |

## 二十一、驗收標準

- 手機可讀，主要文字、按鈕與卡片在常見手機寬度下不擁擠、不溢出。

- 搜尋常用詞可命中，例如生病、請假、早退、午餐、接送、霸凌、性平、失物招領。
- 每張卡片有負責處室與更新日期。
- 敏感議題不做判斷，只提供求助管道、保護原則與學校依法處理方向。
- 無學生個資。

- 連結有效，失效或待補連結不誤導使用者。
